# matching_feedback — wallow validation example

A self-contained walkthrough of the workflow described in `specs/wallow_spec.md` §14: a meta-learning dispatcher that registers candidate runs deduplicated by `(cell_k, cell_sigma, generation, candidate_id, seed, warmup_steps)`, plus a downstream analysis script using the DSL.

The schema in `wallow.toml` is the matching-feedback schema from spec §3 (with `warmup_steps` added in the second migration).

## Layout

```
examples/matching_feedback/
├── wallow.toml                   schema (5 identifying + 10 annotating fields)
├── alembic.ini                   anchored to this directory
├── alembic/
│   ├── env.py
│   ├── script.py.mako
│   ├── versions/
│   │   ├── ..._initial_schema.py
│   │   └── ..._add_warmup_steps.py
│   └── snapshots/
│       ├── <rev1>.toml           wallow.toml as of revision 1
│       └── <rev2>.toml           wallow.toml as of revision 2
├── dispatcher.py                 populates runs.db (resume-safe)
├── analysis.py                   queries via the DSL
└── runs.db                       created by `migrate apply`; gitignored
```

## End-to-end run

From this directory:

```bash
wallow migrate apply         # creates runs.db at head revision
python dispatcher.py         # registers 144 runs across a small grid
python dispatcher.py         # rerun → "0 new, 144 already done"
python analysis.py           # top-k, set-membership, JSON-path, find()
```

The two-migration history is checked in. To inspect it:

```bash
wallow migrate history       # both revisions, current marked with *
wallow status                # current vs head, run count
```

## What each piece demonstrates

**Migration 1 (`initial_schema`)** — autogenerated `runs` table with composite UNIQUE on the five original identifying fields, indexes on every identifying field plus the `indexed = true` annotating fields.

**Migration 2 (`add_warmup_steps`)** — autogenerated `op.batch_alter_table` that (a) adds `warmup_steps INTEGER NOT NULL DEFAULT 0`, (b) drops and recreates `uq_runs_identifying` to include the new column, (c) creates `ix_runs_warmup_steps`. The `server_default = '0'` is what lets the migration apply against a non-empty table — it backfills existing rows. This is the §8.2 "adding an identifying field" workflow end-to-end.

**`dispatcher.py`** — the resume-safety pattern. The first `register(..., on_duplicate="return_existing")` per candidate is the dedup gate; the second `register(..., on_duplicate="overwrite")` writes final metrics. Crashing mid-dispatch and rerunning skips already-completed work.

**`analysis.py`** — boolean composition with parenthesised comparisons (`(F("k") == 4) & (F("status") == "completed")`), `order_by(...).limit(10)`, `in_`, JSON path indexing (`F("discovered_T").json_path("T_44").is_not_null()`), and `find()` for direct identifying-key lookup.
